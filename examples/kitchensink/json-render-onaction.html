JSON Render onaction IDL property

Verifies the React-style onaction property on <json-render>: assigning a function registers a single jr-action listener, reassigning replaces the previous listener without stacking, assigning null removes it, assigning any other non-function value throws a TypeError, declarative onaction=${fn} written in a hyper-element template must wire through the same setter, the auto-busy lock must continue to fire independently, and disconnecting the element must clean up any active listener.

AC #1 — Assigning a function fires on jr-action

element.onaction = fn must register a single jr-action listener. Dispatching the event must invoke fn with the CustomEvent, and the event's detail payload must be reachable.

AC #2 — Reassigning replaces, never stacks

Assigning a second function must remove the prior listener atomically before registering the new one. After a single dispatch, the second function must have fired exactly once and the first must never have fired.

AC #3 — Assigning null removes the listener

After element.onaction = null, dispatching jr-action must not invoke the previously registered handler. Reading element.onaction must return null.

AC #4 — Non-function, non-null values throw TypeError

Strict semantics (chosen over native onclick-style silent coercion): assigning a string, number, object, or boolean must throw TypeError. Any previously registered handler must be left intact — a failed assignment must not clear state as a side effect.

AC #5 — Reassigning the same function is idempotent

Setting element.onaction to a function that is already the active handler must be a no-op. Only one handler must fire on dispatch.

AC #6 — Declarative onaction=${fn} in a template

Inside a hyper-element template, the attribute onaction=${fn} is routed through hyper-element's directFor path, which assigns the property via node.onaction = fn. That must land in our setter and wire up the jr-action listener just like the imperative form.

AC #7 — Auto-busy lock still fires when onaction is set

The host's capture-phase jr-action listener (installed in setup()) is independent from onaction. Both must fire: the busy marker must be stamped, and onaction must still observe the event.

AC #8 — Disconnect removes the active listener

Removing the element from the DOM must trigger its teardown. After reconnection with no new assignment, the previously registered onaction handler must not fire on subsequent dispatches — i.e. the listener does not leak across connect/disconnect cycles.

AC #9 — @jr-action swap does not duplicate listeners

Passing a fresh arrow through @jr-action=${(e) => ...} across successive renders must swap (not stack) the listener. After N renders with N distinct closures, a single dispatch must invoke only the most recent closure, and exactly once.

AC #10 — Same-reference rebind does not accumulate listeners

Observable outcome guaranteed by the combination of Hole.update()'s value !== prev skip (src/render/hole.js) and the prev === value early-return added to event() (src/render/update.js): passing the same function reference across many renders must produce exactly one live listener. Dispatching the target event must invoke the handler exactly once — never zero (the listener survived), never more than one (duplicates stacked).

AC #11 — Plain-DOM pre-connect assignment is hoovered by setup()

When a consumer does jr.onaction = fn before appending the element (no hyper-element template involved), the assignment lands as a plain expando. On connection, setup() captures the expando value, defines the accessor descriptor, then re-applies the value through the accessor. The listener must be wired correctly by the time the first jr-action event fires.

AC #12 — onaction is a non-enumerable accessor

Native on* handlers on DOM elements are non-enumerable: they do not show up in Object.keys(el) or for…in. Our accessor must match that shape so serialization paths (JSON.stringify, Object.keys-driven introspection) do not surface the handler as a field. The property descriptor must also be configurable so teardown can delete it on disconnect.

AC #13 — Capture-phase busy listener runs before onaction

The auto-busy listener installed by setup() is capture-phase, while onaction is bubble-phase (via addEventListener with no third argument). The DOM event model guarantees capture runs before bubble, so by the time the consumer's onaction handler observes the event, data-jr-busy is already stamped on the host — consumers can rely on "lock engaged before my code runs."

AC #14 — Pre-connect typo logs a console error but does not corrupt the element

Strict-typing policy is applied at both assignment sites, but with different surfacing to match browser constraints: post-connect assignments (AC #4) throw TypeError synchronously from the setter; pre-connect typos cannot throw out of appendChild (the spec requires the insertion to continue), and throwing out of setup() would abort the rest of the setup hook and leave the element without a render pipeline. The chosen policy: the setter's throw is caught at the narrow re-apply site in setup() and logged via console.error with the original TypeError message. The element then renders normally and jr.onaction round-trips to null (no listener wired). This preserves loud-surfacing for typos while keeping the UI intact.
