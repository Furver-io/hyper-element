JSON Render Component Bridge

Verifies the jrType + jrCatalog bridge — a hyperElement(...) declaration that auto-registers the custom element into the json-render component registry. Each section maps to one acceptance criterion from TASK-04.

AC #1 — Override built-in via jrType

hyperElement('ac1-card', { jrType: 'Card', jrCatalog, render }) — a spec with type: 'Card' must render the bridged <ac1-card>, NOT the built-in .jr-card.

AC #2 — Mixed spec (overridden Card + built-in Button + custom ProductCard)

Renders a spec containing the AC #1 overridden Card as the root, a built-in Button child, and a custom-only ProductCard child. Every node must resolve to the correct renderer.

AC #3 — jr-action events bubble from bridged components

A bridged custom element dispatches a jr-action event from its own click handler. The listener attached to the <json-render> host must observe it.

AC #4 — data-jr-props attribute carries serialised props

After render, getAttribute('data-jr-props') on the bridged element equals JSON.stringify(def.props).

AC #5 — Children appear as light-DOM AND data-jr-children

At connect time the bridged element has its rendered kids as light-DOM children (captured via setup() before render() runs) AND data-jr-children equals JSON.stringify(def.children).

AC #6 — data-jr-on attribute carries serialised events

After render, getAttribute('data-jr-on') on the bridged element equals JSON.stringify(def.on).

AC #7 — getCatalog() reflects custom catalog metadata

After AC #1's override, getCatalog().types.get('Card') equals the custom catalog object — proving the LLM-facing schema sees the overridden component, not the built-in.

AC #8 — prompt() includes the custom component description and props

getCatalog().prompt() contains the AC #1 custom description and at least one custom prop name.

AC #10b — jrType without a tag name throws a diagnostic error

hyperElement({ jrType: 'Foo', render }) (tagless form) must throw, since the bridge needs document.createElement(tag) to instantiate the bridged element. The error message must name the offending jrType and point at the correct call form.

AC #10 — Component without jrType is unaffected

hyperElement('ac10-plain', { render }) registers a normal custom element. It must NOT appear in listComponentTypes() nor in getCatalog().

AC #11 — jrType without jrCatalog (legacy mode)

hyperElement('ac11-log', { jrType: 'Ac11LogType', render }) — registry contains the render fn so the spec renders, but getCatalog().types does NOT contain the type (no metadata = invisible to LLM).

AC #12 — Partial-spec shimmer still works with bridged components

Empty <json-render></json-render> still renders the loading shimmer placeholder even after several bridge registrations have mutated the registry.

AC #9 — Last-write-wins with console.warn

Two hyperElement definitions with the same jrType: 'Ac9Type' — the second wins and a console.warn fires. Spec rendering produces the SECOND tag's element. Runs LAST so the collision does not contaminate earlier sections.
