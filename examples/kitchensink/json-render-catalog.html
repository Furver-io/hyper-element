JSON Render Catalog API

Verifies getCatalog() — the public catalog API that walks the json-render registry and produces LLM-consumable output. Each section maps to one acceptance criterion from TASK-03.

AC #1 — getCatalog().types lists all 12 built-ins

After getCatalog(), .types must be a Map of size 12 containing every built-in: Card, Row, Column, Button, Text, Alert, Progress, Divider, CodeBlock, Image, Checklist, TextField.

AC #2 — Legacy function-only registrations are excluded

After registerComponent('LegacyFn', () => null), the type must NOT appear in getCatalog().types because it has no catalog metadata.

AC #3 — prompt() contains every component's full metadata

Verifies the natural-language prompt contains: each built-in's name, the formatted prop strings (with required/enum/nullable flags), the Children: yes/no line, and the full Actions: line for Button.

AC #4 — prompt() includes the output-format spec block

The prompt must include the literal { root, elements } format block describing the spec the LLM should emit.

AC #5 — prompt({ customRules }) appends rules at the end

Custom rules must appear under a Rules: heading at the bottom of the prompt, after the output-format block.

AC #6 — toolDefinition() returns valid JSON Schema

toolDefinition({ name: 'render_ui' }) must return a Claude/OpenAI-compatible tool definition with the correct structure and a type.enum listing every cataloged component.

AC #7 — Custom catalog-tagged components appear everywhere

After registerComponent('CatalogThing', { render, catalog }), the new type must appear in .types, in the .prompt() output, and in the .toolDefinition() enum.

AC #8 — Returned snapshot is immutable

The snapshot instance is frozen, each entry is recursively frozen (mutations throw TypeError in strict mode), and a subsequent getCatalog() call returns the unmutated registry view.
