JSON Render Auto-Busy Lifecycle

Verifies the auto-busy interaction lifecycle on <json-render>: dispatching a jr-action from any descendant interactive component must atomically set data-jr-busy="true" on the host, the next textContent replacement must clear it, and the data-jr-busy-mode="off" attribute must opt the element out of the behavior.

AC #1 — A freshly mounted <json-render> has no busy marker

Mounting a spec without dispatching any action must leave the host element with neither data-jr-busy nor data-jr-busy-action set. The marker is purely interaction-driven, never spec-driven.

AC #2 — Dispatching jr-action sets data-jr-busy synchronously

Programmatically clicking the rendered Button must trigger the Button's built-in dispatchAction, which the host listener (installed in capture phase by element.js's setup hook) catches and uses to set data-jr-busy="true" and data-jr-busy-action="approve".

AC #3 — Replacing textContent clears the busy marker

After dispatching an action, replacing the host's textContent with a new spec must trigger the MutationObserver-driven re-render and clear both data-jr-busy and data-jr-busy-action as part of render(). This is what makes the lock auto-release when the LLM responds with a fresh spec.

AC #4 — data-jr-busy-mode="off" disables auto-busy

Setting data-jr-busy-mode="off" on the host element before mounting must prevent the listener from setting data-jr-busy on dispatch. Consumers that want full external control over interaction state use this opt-out.
